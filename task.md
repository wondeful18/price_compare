# price_compare_desktop 开发任务拆解（task.md）

## 1. 文档目标

本文档用于把 [spec.md](/i:/biaoshu/price_compare/spec.md) 和 [arch.md](/i:/biaoshu/price_compare/arch.md) 拆成可逐阶段验收的开发任务。

执行原则：

1. 每次只做一个阶段。
2. 当前阶段未验收通过，不进入下一阶段。
3. 每个阶段都必须有可运行成果，不接受只搭代码骨架但无法验证。
4. 每个阶段都限制范围，避免提前做后续功能。

---

## 2. 总体开发策略

### 2.1 交付方式

本项目按 6 个阶段推进：

- Phase 1：项目骨架 + Excel 导入预览
- Phase 2：单平台搜索主链路跑通
- Phase 3：缓存、导出、基础日志
- Phase 4：AI 增强能力接入
- Phase 5：淘宝接入与成本控制
- Phase 6：结果优化、稳定性与收尾

### 2.2 阶段门禁

每阶段结束后都按以下口径验收：

- 功能是否能实际演示
- 核心路径是否有最少测试
- 代码是否符合 `arch.md` 的分层约束
- 是否出现明显“为了快而写死”的实现

只有以上满足，才进入下阶段。

### 2.3 明确不允许

在前期阶段不允许提前做这些内容：

- 提前接入多个 Provider
- 提前做复杂 AI 决策
- 提前做云端、账号、下单能力
- 提前做过度设计的插件体系
- 提前做真实外部 API 的自动化 CI 依赖

## 4. Phase 1：项目骨架 + Excel 导入预览

### 4.1 阶段目标

先把桌面应用最基本的“看得见、能交互、能导入 Excel、能解析成结构化数据、能表格展示”跑通。

这一阶段不做任何真实搜索。

### 4.2 范围

包含：

- 初始化项目目录结构
- 建立核心数据模型和枚举
- 建立配置加载基础能力
- 建立主窗口和基础 UI 布局
- 实现 Excel 首工作表读取
- 自动识别表头并解析为 `MaterialItem`
- 在界面中展示导入预览

不包含：

- Provider 实现
- 后台搜索线程
- AI
- SQLite 缓存
- 导出 Excel

### 4.3 具体任务

1. 建立项目目录骨架
2. 建立基础依赖和启动入口 `app.py`
3. 定义 `MaterialItem`、`TaskStatus`、`PlatformType` 等核心模型
4. 实现 `excel_reader.py`
5. 实现 `ImportAppService`
6. 完成主界面基础布局
7. 完成结果表格的预览展示
8. 对导入失败给出基础错误提示

### 4.4 交付物

- 可启动桌面程序
- 导入按钮可选择本地 `.xlsx`
- 成功读取首个工作表
- 能把行数据映射成 `MaterialItem`
- 中央表格可展示导入结果
- 至少有 1 个示例 Excel 可用于演示

### 4.5 验收标准

满足以下条件视为通过：

1. 启动应用后可以看到完整主窗口。
2. 点击“导入 Excel”可以读取 `.xlsx` 文件。
3. 至少能识别这些字段中的大部分：序号、名称、规格、数量、单位、品牌、采购价原文。
4. 导入后表格中至少能正确显示 20 行以上数据。
5. 文件不存在、文件格式错误、工作表为空时，界面不会崩溃。
6. UI 代码没有直接写 Excel 解析逻辑，解析在应用层/基础设施层。

### 4.6 最低测试要求

- `excel_reader` 单元测试
- `ImportAppService` 单元测试
- 表头识别的至少 2 组样例测试

### 4.7 通过后才能进入下一阶段的前置条件

- 项目目录结构稳定
- 物料数据模型稳定
- 导入和预览链路可反复演示

## 5. Phase 2：单平台搜索主链路跑通

### 5.1 阶段目标

把“从一条物料生成查询词，到调用一个平台搜索，再到候选打分和结果落表”的核心业务链跑通。

这一阶段只接一个主 Provider：`JDOfficialProvider`。

### 5.2 范围

包含：

- `SearchQuery`、`ProductOffer`、`CompareResult` 模型
- 查询词规则生成
- `JDOfficialProvider` 接口实现
- 搜索线程模型
- 批量搜索任务调度
- 候选结果标准化、打分、排序
- 结果回写 UI 表格

不包含：

- 淘宝 Provider
- AI 优化
- SQLite 缓存
- 导出 Excel

### 5.3 具体任务

1. 实现 `QueryBuilderService`
2. 实现 `SearchProvider` 抽象和 `BaseProvider`
3. 实现 `JDOfficialProvider`
4. 实现统一 `http_client.py`
5. 实现 `SearchTaskService`
6. 实现 `TaskRunner`、`TaskBus`
7. 实现候选评分模块 `scoring.py`
8. 在结果表中显示推荐平台、推荐价格、状态
9. 支持开始搜索和停止任务

### 5.4 交付物

- 导入后的物料可以发起搜索
- 至少支持单条和批量触发搜索
- JD 返回结果能标准化为 `ProductOffer`
- 每条物料最终有 `CompareResult`
- UI 实时显示“未处理/搜索中/完成/失败”

### 5.5 验收标准

满足以下条件视为通过：

1. 可以对导入后的物料执行搜索。
2. UI 搜索过程中不会卡死。
3. 至少 10 条物料批量搜索时，状态能持续刷新。
4. 每条完成后的记录，表格中可看到推荐平台和推荐价格。
5. 某条搜索失败时，不会导致整个批次中断。
6. Worker 线程没有直接更新 UI，而是通过事件回传。
7. Provider 逻辑没有散落在 UI 或应用入口里。

### 5.6 最低测试要求

- `query_builder` 单元测试
- `scoring` 单元测试
- `JDOfficialProvider` 响应解析测试
- `SearchTaskService` 事件流测试

### 5.7 通过后才能进入下一阶段的前置条件

- JD 搜索主链路可稳定演示
- 结果表字段已经基本定型
- 线程模型已经可用

## 6. Phase 3：缓存、导出、基础日志

### 6.1 阶段目标

把产品从“能搜”提升到“本地可实际使用”，补齐缓存、导出、日志这三个 V1 的基础能力。

### 6.2 范围

包含：

- SQLite 初始化
- 搜索缓存读写
- 缓存命中统计
- 结果导出 Excel
- 基础日志
- 搜索结果和导出结果的本地持久化

不包含：

- AI
- 淘宝 Provider
- 预算策略

### 6.3 具体任务

1. 实现 `sqlite_db.py`
2. 实现 `cache_repository.py`
3. 完成 `search_cache` 表初始化
4. 在搜索主链路接入缓存
5. 实现 `ExportAppService`
6. 实现 `excel_writer.py`
7. 导出时保留原始列并追加结果列
8. 增加 `loguru` 日志
9. UI 增加缓存命中和导出成功提示

### 6.4 交付物

- 命中缓存时不再重复请求 JD
- 可以导出带结果的新 Excel
- 导出的文件包含原始数据、推荐价、推荐平台、差价、状态
- 产生本地日志文件

### 6.5 验收标准

满足以下条件视为通过：

1. 相同查询第二次执行时，能够明显走缓存路径。
2. 表格中能区分“使用缓存”和“实时搜索”。
3. 点击“导出结果”后生成新的 `.xlsx` 文件。
4. 导出文件保留原始列，不覆盖原始数据。
5. 导出文件中新增至少这些列：推荐平台、推荐价、差价、状态。
6. 搜索和导出过程会写本地日志。
7. 缓存逻辑不写死在 Provider 内部，而由仓储/服务层接入。

### 6.6 最低测试要求

- `cache_repository` 单元测试
- 缓存 key 生成测试
- `ExportAppService` 集成测试
- Excel 导出列完整性测试

### 6.7 通过后才能进入下一阶段的前置条件

- 单平台闭环已经具备重复使用价值
- 导入 -> 搜索 -> 导出链路已完整

## 7. Phase 4：AI 增强能力接入

### 7.1 阶段目标

在不破坏主流程稳定性的前提下，把 AI 作为“增强层”接入，而不是让 AI 接管整个比价决策。

### 7.2 范围

包含：

- `DeepSeek` 客户端封装
- AI 查询优化
- AI 候选解释
- AI 缓存
- 空结果后二次重搜策略

不包含：

- 淘宝 Provider
- 复杂预算联动
- AI 自动决策最终结果

### 7.3 具体任务

1. 实现 `deepseek_client.py`
2. 实现 `AiService`
3. 定义 AI Prompt 和结构化返回格式
4. 实现 AI 缓存表 `ai_cache`
5. 在 `QueryBuilderService` 中增加 AI 优化开关
6. 在 `CompareAppService` 中增加 AI 解释生成
7. 在空结果场景增加“AI 改写后重搜一次”
8. UI 增加 AI 开关和 AI 备注列

### 7.4 交付物

- 配置 API Key 后可以启用 AI
- AI 能优化复杂物料的搜索词
- AI 能为推荐候选生成解释
- AI 调用失败时主流程仍可继续

### 7.5 验收标准

满足以下条件视为通过：

1. AI 开启后，复杂物料能生成优化后的关键词。
2. 空结果场景下可进行一次 AI 改写重搜。
3. 表格或详情面板中能看到 AI 解释文本。
4. AI 服务失败、超时或返回异常时，批量任务不会崩溃。
5. AI 结果有缓存，不会对相同输入重复请求。
6. AI 仅作为增强，不直接绕过 Provider 给最终价格。

### 7.6 最低测试要求

- `AiService` 单元测试
- AI 输出解析测试
- AI 缓存测试
- 空结果重搜流程测试

### 7.7 通过后才能进入下一阶段的前置条件

- AI 能稳定降级
- 主搜索链路没有被 AI 耦死

## 8. Phase 5：淘宝接入与成本控制

### 8.1 阶段目标

把系统从“单平台搜索工具”升级为“多平台比价工具”，同时加入预算和调用成本控制。

### 8.2 范围

包含：

- `TaobaoThirdPartyProvider`
- 多 Provider 编排
- 成本统计
- 预算阈值
- 高价值物料补搜策略
- 平台启用开关

不包含：

- 拼多多
- WebFallbackProvider
- 复杂历史价格分析

### 8.3 具体任务

1. 实现 `TaobaoThirdPartyProvider`
2. 扩展 `SearchTaskService` 支持多平台
3. 实现 `CostGuardService`
4. 实现调用次数统计
5. 实现预估成本统计
6. 实现预算达到阈值后的停止补搜
7. 支持按平台开关搜索
8. 在 UI 中展示各平台价格和任务预算信息

### 8.4 交付物

- 京东和淘宝两平台可并行/顺序执行
- 表格中可显示京东价、淘宝价、推荐平台、推荐价
- 任务区可看到调用次数和预算信息
- 超预算时系统会降级策略

### 8.5 验收标准

满足以下条件视为通过：

1. 可以在设置里启用/禁用淘宝。
2. 两平台结果都能统一映射成 `ProductOffer`。
3. 同一条物料可以得到跨平台推荐结果。
4. 系统能显示总调用次数、各平台调用次数、缓存命中次数、预估成本。
5. 超预算后会停止补搜，而不是继续无控制调用。
6. 低价值物料和高价值物料的搜索策略有差异化。
7. 替换淘宝供应商时不需要改动主搜索流程。

### 8.6 最低测试要求

- `TaobaoThirdPartyProvider` 响应解析测试
- `CostGuardService` 单元测试
- 多 Provider 结果整合测试
- 超预算策略测试

### 8.7 通过后才能进入下一阶段的前置条件

- 双平台链路可稳定演示
- 预算控制已可用
- 平台扩展方式已验证

## 11. 每阶段输出要求

为了避免阶段间反复返工，每一阶段开发结束时都应额外输出一份简短总结，至少包含：

- 本阶段完成内容
- 未完成内容

- 是否建议进入下一阶段

---

## 12. 当前建议结论

建议按以下顺序推进：

1. Phase 1 先把导入和预览做稳。
2. Phase 2 只打一条主平台链路，不扩散。
3. Phase 3 补齐缓存和导出，形成最小可用闭环。
4. Phase 4 再引入 AI，确保 AI 只是增强层。
5. Phase 5 再加淘宝和成本控制，避免前期复杂度过高。
6. Phase 6 最后统一做稳定性和交付收尾。

这个顺序的核心是先保闭环，再保扩展，再做增强，能最大程度降低返工。
